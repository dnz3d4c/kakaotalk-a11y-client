# AI 프롬프트 활용 예시

카카오톡 접근성 클라이언트를 개발하면서 실제로 사용한 프롬프트를 정리한 문서입니다.
각 프롬프트의 사용 상황, 예시, 결과, 분석(좋은 점 / 보완할 점)을 담았습니다.

Claude Code용 개발 프롬프트는 `docs/카카오톡_*.md` 파일을 참고하세요.

---

## 개요

| # | 패턴 | 사용 상황 |
|---|------|----------|
| 1 | 이미지/스크린샷 분석 | 스크린 리더로 접근 불가한 UI 구조 파악 |
| 2 | 기술 타당성 검토 | 새 기능 개발 전 여러 방법 비교 |
| 3 | 도구/라이브러리 적합성 분석 | 새 도구 도입 전 목적 부합 여부 확인 |
| 4 | 기술 개념 설명 요청 | 모르는 개념을 프로젝트 맥락에 맞게 이해 |
| 5 | Claude Code 프롬프트 개선 조언 | 작성한 프롬프트 품질 검토 |
| 6 | 작업 계획 수립 | 복잡한 작업을 체계적으로 진행 |
| 7 | 리서치 요청 | 공식 문서 없는 정보 수집 |

## 1. 이미지/스크린샷 분석

### 사용 상황

카카오톡 공감 이모지가 스크린 리더로 인식되지 않아, 일단 이게 뭔지 눈으로 보는 사람한테 물어보듯이 Claude한테 물어봤다.

### 프롬프트

```
지금 붙여넣은 이미지에 뭐가 보이는지 설명해
```

```
오 좋아 그 이모지 반응에 집중해줘.
이 이모지 반응이 무엇인지 설명해줄래?
```

```
혹시 이모지 반응한 숫자도 카운트된 게 인식돼?
```

### 결과

Claude가 스크린샷에서 다음을 파악:
- UI 요소의 위치와 배치
- 텍스트 내용
- 아이콘/이모지 종류와 순서
- 시각적 구분 (색상, 크기 등)

### 프롬프트 분석

**좋은 점**:
- 단계적으로 질문함 (전체 → 특정 영역 → 세부 정보)
- 자연스러운 대화체로 follow-up
- "오 좋아"로 피드백 주면서 방향 유도

**보완할 점**:
- 첫 질문에 "카카오톡 채팅창 스크린샷이야" 맥락 추가하면 더 정확했을 것
- 목적("공감 이모지 클릭 프로그램 만들 거야")을 미리 말했으면 더 유용한 정보 얻었을 수 있음

## 2. 기술 타당성 검토

### 사용 상황

공감 이모지 UI 구조를 파악한 후, 실제로 클릭하는 프로그램을 만들 수 있는지 검토받고 싶었다. 혼자 판단하기 어려워서 여러 방법을 비교해달라고 요청했다.

### 프롬프트

```
이 이모지를 클릭하는 프로그램을 만들고 싶은데, 가능 여부를 다각도로 검토해줘.
현재 스크린 리더로는 이 이모지에 접근할 방법이 없어.
```

### 결과

Claude가 표 형태로 여러 접근 방법 비교:

| 방법 | 실현 가능성 | 장점 | 단점 |
|------|-------------|------|------|
| UI Automation API | 낮음 | 안정적 | 접근성 정보 노출 안 하면 무용지물 |
| 이미지 인식 + 좌표 클릭 | 중간 | 접근성 정보 없어도 동작 | 창 크기 변경 시 깨짐 |
| 고정 좌표 클릭 | 낮음 | 구현 단순 | 불안정 |
| 내부 API 후킹 | 기술적 가능 | 정확한 제어 | 약관 위반, 난이도 높음 |

### 프롬프트 분석

**좋은 점**:
- 목적 명확함 ("이모지를 클릭하는 프로그램")
- 현재 제약 조건 명시함 ("스크린 리더로 접근 불가")
- "다각도로 검토해줘"로 여러 대안 요청

**보완할 점**:
- "100% 로컬에서만 동작해야 함" 같은 제약 조건 추가했으면 더 좁혀졌을 것
- "표로 비교해줘" 명시했으면 바로 표로 받았을 수 있음

## 3. 도구/라이브러리 적합성 분석

### 사용 상황

Accessibility Insights라는 도구를 발견하고, 지금 개발에 도움이 될지 검토받고 싶었다.

### 프롬프트

```
https://accessibilityinsights.io/docs/windows/overview/
이 도구를 써보려고 하는데, 지금 상황에서 적절한지 검토해줘.
자동화 테스트나, UI 말고 API로 테스트 가능한지 개발에 도움을 얻을 수 있는지 다각도에서 심층 분석해 알려줘.
```

### 결과

Claude가 목적 비교표 제공:

| 항목 | 우리 목적 | 도구 목적 |
|------|----------|-----------|
| 핵심 | UIA 구조 파악 → 조작 | 접근성 규칙 준수 검증 |
| 대상 앱 | 커스텀 UI 앱 | 표준 프레임워크 앱 |
| 결과 활용 | 탐색 경로, 패턴 파악 | WCAG 위반 리포트 |

결론: "목적이 다름" → 불필요한 도구 도입 방지

### 프롬프트 분석

**좋은 점**:
- URL 직접 제공해서 Claude가 바로 확인 가능
- 구체적인 질문 ("API로 테스트 가능한지")

**보완할 점**:
- "심층 분석" 대신 "우리 목적과 맞는지 비교해줘"가 더 명확했을 것

## 4. 기술 개념 설명 요청

### 사용 상황

개발 중에 "가상 스크롤" 때문에 문제가 생겼는데, 이게 뭔지 몰랐다. 단순 정의보다 지금 프로젝트에서 왜 문제인지 맥락에 맞게 설명받고 싶었다.

### 프롬프트

```
UIA의 가상 스크롤 개념이 뭔지 설명해줄래? 
카톡 접근성 클라이언트 개발에 필요한 이유는?
```

### 결과

Claude가 개념 + 실제 문제 + 해결법까지 연결해 설명:

**가상 스크롤이란**: 화면에 보이는 요소만 실제 생성, 나머지는 placeholder

**카카오톡에서 문제**:
- 메시지 10개인데 ListItem은 174개
- 빈 항목 164개 (94%)
- 탐색 느려짐, 인덱스 꼬임

**해결법**: 조기 종료 패턴, 유효 범위 캐싱

### 프롬프트 분석

**좋은 점**:
- 단순 정의 요청이 아니라 프로젝트 맥락 연결 ("카톡 접근성 클라이언트 개발에 필요한 이유")
- 짧고 명확함

**보완할 점**:
- "지금 겪고 있는 문제: 메시지 10개인데 ListItem이 174개 나옴" 같이 증상을 먼저 말했으면 더 구체적인 답변 받았을 것

## 5. Claude Code 프롬프트 개선 조언

### 사용 상황

Claude Code한테 작업 지시할 프롬프트를 작성했는데, 이게 잘 쓴 건지 모르겠어서 Claude한테 검토받았다.

### 프롬프트

```
클로드용 프롬프트 문서 읽고 Phase 1 (profiler.py)부터 순서대로 구현해.
각 단계 완료 후 간단히 테스트하고 커밋해.
이렇게 클로드 코드에 프롬프트 주려 하는데 개선 포인트 있어?
```

### 결과

Claude가 문제점과 개선안 제시:

| 항목 | 기존 | 개선 |
|------|------|------|
| 문서 참조 | "문서 읽고" | "첨부한 파일명 읽어" |
| 파일 경로 | 없음 | 정확한 경로 명시 |
| 테스트 방법 | "간단히" | 구체적 명령어 + 확인 항목 |
| 진행 조건 | 없음 | "완료 후 결과 알려줘" |
| 에러 대응 | 없음 | "에러 시 메시지 보여주고 수정" |

### 개선된 프롬프트

```markdown
# 카카오톡 접근성 클라이언트 개선 작업

## 참고 문서
첨부한 "카카오톡_접근성_개선_프롬프트.md" 파일 읽어.

## 작업 순서
Phase 1부터 순서대로 진행:

### Phase 1: 진단 도구
1. `src/kakaotalk_a11y_client/utils/profiler.py` 생성
2. 기존 `uia_utils.py`, `context_menu.py`에 프로파일링 적용
3. 테스트: `uv run python -m kakaotalk_a11y_client.main` 실행 → 에러 없이 시작되는지 확인
4. 커밋

## 규칙
- 각 Phase 완료 후 나한테 결과 알려줘
- 에러 발생 시 에러 메시지 전체 보여주고 수정 시도
```

### 프롬프트 분석

**좋은 점**:
- 메타적 접근 (프롬프트에 대한 프롬프트)
- 구체적인 예시를 보여주면서 질문

**보완할 점**:
- 원본 프롬프트가 뭐가 문제인지 먼저 분석했으면 더 날카로운 피드백 받았을 것

## 6. 작업 계획 수립

### 사용 상황

UIA에 대해 잘 몰라서 문서화하고 싶었는데, 어디서부터 시작해야 할지 막막했다. 예전에 Qt 접근성 문서 작성한 경험이 있어서 그 구조를 참고하라고 했다.

### 프롬프트

```
내가 UIA에 대해 잘 몰라.
구현한 기능이 동작할 때와 동작하지 않을때를 예측하기 어려워
예전에 Qt 접근성 관련 문서 작성했던 거 기억하지?
참고해서 UIA 관련한 문서 작성해봐.
어떤 문서를 작성했는지는 이전 대화 참고하고, 작성 전 작성 계획을 세워.
```

### 결과

Claude가 과거 대화 검색 후 구조화된 계획 제시:

**Qt 문서 구조 (참고용)**:
- 이론/개념 4개
- 실전/레퍼런스 5개
- 추가 3개

**UIA 문서 계획**:

| # | 문서명 | 내용 |
|---|--------|------|
| 1 | UIA 아키텍처 개괄 | Provider/Client 구조, MSAA와 차이 |
| 2 | UIA 핵심 개념 | ControlType, Pattern, Property |
| 3 | Python UIA 패키지 비교 | uiautomation vs pywinauto |
| ... | ... | ... |

**추가 리서치 필요 여부**도 함께 제시

### 프롬프트 분석

**좋은 점**:
- 이전 대화 참조 활용 ("예전에 Qt 접근성 관련 문서 작성했던 거 기억하지?")
- 현재 문제 상황 공유 ("구현한 기능이 동작할 때와 동작하지 않을때를 예측하기 어려워")
- 단계적 접근 요청 ("작성 전 계획 세워")

**보완할 점**:
- "문서 10개 이내로" 같은 범위 제한 줬으면 더 현실적인 계획 나왔을 것

## 7. 리서치 요청

### 사용 상황

공감 이모지 클릭 기능을 개발하기 전에, 카카오톡 공감 기능 스펙을 정확히 알고 싶었다. 공식 문서가 없어서 Claude한테 웹 검색해서 정리해달라고 했다.

### 프롬프트

```
개발하기 전 카카오톡 말풍선 공감에 대해 리서치를 수행해 찾아보고 현재 상황을 보완해봐.
적용 가능한 방법이 있다면 클로드 코드에서 내가 개발 시도해볼게.
```

### 결과

Claude가 웹 검색 후 정리:

**기본 스펙**:
- 출시일: 2021년 8월 4일
- 이모지 종류: 6종 (하트, 엄지, 체크, 웃음, 놀람, 슬픔)
- 제한: 7일 지난 메시지에는 사용 불가

**접근성 현황**:
- 스크린 리더 접근: 미지원
- 키보드 네비게이션: 미지원

**개발 시 고려사항**:
- 이모지 6개 고정 → 템플릿 6개만 준비
- 라이트모드/다크모드 분기 필요

### 프롬프트 분석

**좋은 점**:
- 후속 작업 연결 명시 ("적용 가능한 방법이 있다면 클로드 코드에서 개발")

**보완할 점**:
- "이모지 종류, 사용 제한, 접근성 지원 현황" 같이 알고 싶은 항목 구체적으로 나열했으면 빠짐없이 받았을 것
- "표로 정리해줘" 추가했으면 결과물 형태 예측 가능했을 것

## 프롬프트 작성 팁

위 예시들에서 공통으로 발견되는 패턴:

### 1. 맥락 먼저 제공
```
지금 PC 카톡 클라이언트를 만드는데, [구체적 상황]
```

### 2. 원하는 결과 형태 명시
```
다각도에서 심층 분석해 알려줘
표로 비교해줘
작업 전 계획 세워
```

### 3. 제약 조건 명확히
```
로컬에서만 사용되야 함
외부 유출이 안 되야 해
```

### 4. 후속 작업 연결
```
... 되면 클로드 코드에서 개발할 거야
이 결과로 프롬프트 만들어줘
```

### 5. 이전 대화 참조 활용
```
예전에 Qt 접근성 관련 문서 작성했던 거 기억하지? 참고해서
```
